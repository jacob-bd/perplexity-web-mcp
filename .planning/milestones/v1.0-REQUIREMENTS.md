# Requirements: Perplexity Web MCP Tool Calling

**Created:** 2026-02-04
**Milestone:** v1 - Best-Effort Tool Calling

## v1 Requirements

### Tool Calling Infrastructure

- [x] **TC-01**: User can provide tool definitions that are injected into prompts using training-data-familiar formats (Python/markdown code blocks) — ✅ Validated in Phase 1
- [x] **TC-02**: System parses responses with multi-strategy fallback chain (JSON extraction → regex patterns → fuzzy matching) — ✅ Validated in Phase 2 (AST, key-value, inline code strategies)
- [x] **TC-03**: System scores extracted tool calls with confidence threshold before execution — ✅ Validated in Phase 4 (0.7 threshold, INFO-level logging)
- [x] **TC-04**: User receives tool execution results injected back into conversation for continuation — ✅ Validated in Phase 4 (format_tool_results, result injection)

### Format Support

- [x] **FMT-01**: System recognizes Python function call format in markdown code blocks (e.g., `search("query")`) — ✅ Validated in Phase 2 (python_ast strategy, confidence: 0.9)
- [x] **FMT-02**: System recognizes simple key-value format (e.g., `SEARCH: query`) — ✅ Validated in Phase 2 (key_value strategy, confidence: 0.7)
- [x] **FMT-03**: System detects inline code patterns mentioning tool calls — ✅ Validated in Phase 2 (inline_code strategy, confidence: 0.5)

### API Compatibility

- [x] **API-01**: POST /v1/messages accepts `tools` array in request body — ✅ Validated in Phase 3
- [x] **API-02**: Response includes `tool_use` content blocks when tool calls are detected — ✅ Validated in Phase 3 (streaming and non-streaming)
- [x] **API-03**: POST /v1/messages accepts `tool_result` messages for continuation — ✅ Validated in Phase 3 (validation and pairing)

### Reliability & Fallbacks

- [x] **REL-01**: System gracefully falls back to conversational response when tool calling fails — ✅ Validated in Phase 5 (classify_parse_error, graceful degradation)
- [x] **REL-02**: System logs format success/failure rates for iteration — ✅ Validated in Phase 5 (Prometheus metrics, /metrics endpoint)
- [x] **REL-03**: System provides clear error messages when tool parsing fails — ✅ Validated in Phase 5 (user-friendly error messages, no stack traces)

## v2 Requirements (Deferred)

- [ ] NLP-based intent extraction using spaCy for conversational responses
- [ ] Multi-turn fallback for extracting tool parameters
- [ ] Model-specific prompt templates (GPT vs Claude vs Gemini)
- [ ] Advanced parameter validation with JSON schema
- [ ] Retry logic with format variation
- [ ] Tool call streaming (partial tool calls as they're generated)

## Out of Scope

- **Intermediate parsing model** — User preference: avoid latency/complexity of second model
- **100% guaranteed success** — Research confirms this is not achievable with prompt engineering
- **Official Perplexity API integration** — Defeats purpose of web interface access
- **Browser automation** — Too fragile for production

## Success Criteria

Given the fundamental limitation (training objective mismatch), success is defined as:

1. **Simple tool calls work reliably** (40-60% success rate) — ✅ Achieved
   - Single-parameter tools (search, read_file)
   - Clear, unambiguous user requests

2. **Complex scenarios fail gracefully** — ✅ Achieved
   - Multi-tool requests return conversational response
   - Ambiguous requests don't cause crashes

3. **System is observable** — ✅ Achieved
   - Success/failure rates logged
   - Easy to identify which formats work best
